## 📊 4.2.1 Equivalence Partitioning (EP) `[K3]`

*FL-4.2.1: Use equivalence partitioning to derive test cases*

### Definition
**Equivalence Partitioning (EP)** divides data into partitions (equivalence partitions) based on the expectation that all elements of a given partition are processed **in the same way** by the test object.

### Key Principle
> If a test case testing one value from a partition detects a defect, this defect should also be detected by test cases testing any other value from the same partition. Therefore, **one test for each partition is sufficient**.

### Partition Characteristics
- Can be identified for **any data element**: inputs, outputs, configuration items, internal values, time-related values, interface parameters
- May be **continuous or discrete**, **ordered or unordered**, **finite or infinite**
- Partitions must **not overlap** and must be **non-empty sets**

### Types of Partitions

| Type | Description |
|------|-------------|
| **Valid Partition** | Contains values that should be processed by the test object |
| **Invalid Partition** | Contains values that should be ignored or rejected by the test object |

### Coverage Calculation
- **Coverage items:** Equivalence partitions
- **100% coverage:** Test cases must exercise all identified partitions (including invalid partitions)
- **Formula:** (Partitions exercised by at least one test case ÷ Total partitions) × 100%

### Each Choice Coverage
For test items with multiple sets of partitions (e.g., multiple input parameters):
- **Each Choice coverage** requires test cases to exercise each partition from each set at least once
- Does NOT take into account combinations of partitions

## 📏 4.2.2 Boundary Value Analysis (BVA) `[K3]`

*FL-4.2.2: Use boundary value analysis to derive test cases*

### Definition
**Boundary Value Analysis (BVA)** is a test technique based on exercising the **boundaries of equivalence partitions**.

### Key Requirements
- Can only be used for **ordered partitions**
- The minimum and maximum values of a partition are its **boundary values**
- If two elements belong to the same partition, all elements between them must also belong to that partition

### Why BVA Works
Developers are more likely to make errors with boundary values. Typical defects found:
- Implemented boundaries **misplaced** above or below intended positions
- Boundaries **omitted altogether**

### Two Versions of BVA

| Version | Coverage Items per Boundary | Description |
|---------|----------------------------|-------------|
| **2-value BVA** | 2 items | Boundary value + its closest neighbor in adjacent partition |
| **3-value BVA** | 3 items | Boundary value + both its neighbors |

### Example: Boundary at x ≤ 10

| Version | Test Values |
|---------|-------------|
| **2-value BVA** | 10, 11 |
| **3-value BVA** | 9, 10, 11 |

### Why 3-value BVA is More Rigorous
If `if (x ≤ 10)` is incorrectly implemented as `if (x = 10)`:
- **2-value BVA** (x = 10, x = 11) will NOT detect the defect
- **3-value BVA** (x = 9) is likely to detect it

## 📋 4.2.3 Decision Table Testing `[K3]`

*FL-4.2.3: Use decision table testing to derive test cases*

### Definition
Decision tables test the implementation of requirements that specify how **different combinations of conditions result in different outcomes**.

### When to Use
- Testing **complex logic** such as business rules
- When multiple conditions affect the outcome

### Decision Table Structure

| Component | Description |
|-----------|-------------|
| **Conditions** | Input conditions (rows) |
| **Actions** | Resulting actions (rows) |
| **Rules** | Columns representing unique combinations of conditions with associated actions |

### Notation

| Symbol | Meaning |
|--------|---------|
| **T** | True - condition is satisfied |
| **F** | False - condition is not satisfied |
| **–** | Irrelevant - value doesn't affect action outcome |
| **N/A** | Infeasible - condition is not applicable for this rule |
| **X** | Action should occur |
| **Blank** | Action should not occur |

### Types of Decision Tables

| Type | Description |
|------|-------------|
| **Limited-entry** | All condition/action values are Boolean (true/false) |
| **Extended-entry** | Conditions/actions may have multiple values (ranges, equivalence partitions, discrete values) |
| **Full** | Covers every combination of conditions |
| **Simplified** | Infeasible combinations deleted |
| **Minimized** | Columns merged where conditions don't affect outcome |

### Coverage Calculation
- **Coverage items:** Columns containing feasible combinations of conditions
- **100% coverage:** Test cases must exercise all feasible columns
- **Formula:** (Exercised columns ÷ Total feasible columns) × 100%

### Strengths and Limitations
- **Strength:** Systematic approach to identify all combinations; finds gaps and contradictions in requirements
- **Limitation:** Number of rules grows exponentially with conditions; may need minimization or risk-based approach

## 🔄 4.2.4 State Transition Testing `[K3]`

*FL-4.2.4: Use state transition testing to derive test cases*

### Definition
State transition testing models the behavior of a system by showing its **possible states and valid state transitions**.

### Key Concepts

| Concept | Description |
|---------|-------------|
| **State** | A condition the system can be in |
| **Transition** | Transfer of control between states; initiated by an event |
| **Event** | Trigger that causes a transition |
| **Guard Condition** | Additional condition that must be true for transition to occur |
| **Action** | Activity performed during transition |

### Transition Labeling Syntax
```
event [guard condition] / action
```
Guard conditions and actions can be omitted if not relevant.

### State Diagram vs. State Table

| Representation | Description |
|----------------|-------------|
| **State Diagram** | Visual model showing states and transitions |
| **State Table** | Rows = states, Columns = events, Cells = target state + actions; explicitly shows invalid transitions (empty cells) |

### Coverage Criteria

| Coverage Type | Coverage Items | Description |
|---------------|----------------|-------------|
| **All States Coverage** | States | All states must be exercised |
| **Valid Transitions Coverage** (0-switch) | Valid transitions | All valid transitions must be exercised |
| **All Transitions Coverage** | All transitions (valid + invalid) | All valid transitions exercised + attempt to execute invalid transitions |

### Coverage Hierarchy
- **All States** < **Valid Transitions** < **All Transitions**
- Achieving full valid transitions coverage guarantees full all states coverage
- Achieving full all transitions coverage guarantees both

> **All transitions coverage** should be a minimum requirement for mission and safety-critical software.

### Best Practice for Invalid Transitions
Test only **one invalid transition per test case** to avoid defect masking (one defect preventing detection of another).

## 🎯 Exam Prep Tips

- **EP `[K3]`:** Know how to identify partitions and that one test per partition is sufficient; understand valid vs. invalid partitions.
- **BVA `[K3]`:** Know 2-value vs. 3-value BVA; understand why 3-value is more rigorous; only works for ordered partitions.
- **Decision Tables `[K3]`:** Know the notation (T, F, –, N/A, X); understand limited-entry vs. extended-entry; know coverage calculation.
- **State Transition `[K3]`:** Know the three coverage criteria and their hierarchy; understand state diagrams vs. state tables.
- **K3 Level:** Be prepared to APPLY these techniques to derive test cases in exam questions.
