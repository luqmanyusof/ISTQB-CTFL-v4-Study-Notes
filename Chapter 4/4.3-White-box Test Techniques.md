## 🔎 Overview

This section focuses on two code-related white-box test techniques:
- **Statement Testing**
- **Branch Testing**

> More rigorous white-box techniques exist for safety-critical, mission-critical, or high-integrity environments but are not covered in this syllabus.

---

## 📝 4.3.1 Statement Testing and Statement Coverage `[K2]`

*FL-4.3.1: Explain statement testing*

### Definition
In **statement testing**, the coverage items are **executable statements**. The aim is to design test cases that exercise statements in the code until an acceptable level of coverage is achieved.

### Coverage Calculation
- **Formula:** (Statements exercised by test cases ÷ Total executable statements) × 100%

### What 100% Statement Coverage Ensures
- All executable statements in the code have been **exercised at least once**
- Each statement with a defect will be executed, which **may cause a failure** demonstrating the defect

### Limitations of Statement Coverage
- Will **NOT detect all defects**, for example:
  - **Data-dependent defects:** Division by zero that only fails when denominator is set to zero
  - **Decision logic defects:** Does not ensure all branches have been tested

> 100% statement coverage does NOT ensure that all decision logic has been tested.

## 🔀 4.3.2 Branch Testing and Branch Coverage `[K2]`

*FL-4.3.2: Explain branch testing*

### Definition
A **branch** is a transfer of control between two nodes in the control flow graph, showing possible sequences in which source code statements are executed.

### Types of Branches

| Type | Description |
|------|-------------|
| **Unconditional** | Straight-line code; always executed |
| **Conditional** | Decision outcome; depends on condition evaluation |

### Coverage Calculation
- **Coverage items:** Branches (both unconditional and conditional)
- **Formula:** (Branches exercised by test cases ÷ Total branches) × 100%

### What 100% Branch Coverage Ensures
- All branches in the code (unconditional and conditional) are exercised by test cases
- Conditional branches typically correspond to:
  - True or false outcome from an `if...then` decision
  - Outcome from a `switch/case` statement
  - Decision to exit or continue in a loop

### Limitations of Branch Coverage
- Will **NOT detect all defects**, for example:
  - Defects requiring execution of a **specific path** in code

### Key Relationship: Branch Coverage Subsumes Statement Coverage

> **Branch coverage subsumes statement coverage:** Any set of test cases achieving 100% branch coverage also achieves 100% statement coverage (but NOT vice versa).

This means:
- 100% branch coverage → 100% statement coverage ✓
- 100% statement coverage → 100% branch coverage ✗

## 💎 4.3.3 The Value of White-box Testing `[K2]`

*FL-4.3.3: Explain the value of white-box testing*

### Fundamental Strength
The **entire software implementation is taken into account** during testing, which facilitates defect detection even when:
- Software specification is **vague**
- Software specification is **outdated**
- Software specification is **incomplete**

### Corresponding Weakness
If the software **does not implement one or more requirements**, white-box testing may **NOT detect the resulting defects of omission**.

### Additional Uses of White-box Techniques
- Can be used in **static testing** (e.g., dry runs of code)
- Well suited for reviewing code **not yet ready for execution**
- Can be applied to **pseudocode** and other high-level or top-down logic modeled with a control flow graph

### Why White-box Coverage Matters
- Performing **only black-box testing** does NOT provide a measure of actual code coverage
- White-box coverage measures provide:
  - **Objective measurement** of coverage
  - **Necessary information** to generate additional tests to increase coverage
  - **Increased confidence** in the code

## 🧩 Key Takeaways

| Technique | Coverage Items | Subsumes |
|-----------|----------------|----------|
| **Statement Testing** | Executable statements | — |
| **Branch Testing** | Branches (unconditional + conditional) | Statement coverage |

| Aspect | Statement Coverage | Branch Coverage |
|--------|-------------------|-----------------|
| **Focus** | Lines of code executed | Decision paths taken |
| **Rigor** | Less rigorous | More rigorous |
| **Relationship** | Subsumed by branch coverage | Subsumes statement coverage |

## 🎯 Exam Prep Tips

- **Statement Testing `[K2]`:** Know that coverage items are executable statements; understand limitations (doesn't test all decision logic).
- **Branch Testing `[K2]`:** Know that coverage items are branches; understand conditional vs. unconditional branches.
- **Subsumption `[K2]`:** Remember that branch coverage subsumes statement coverage (100% branch → 100% statement, but NOT vice versa).
- **Value `[K2]`:** Know the strength (tests entire implementation) and weakness (may miss defects of omission).
- **Static Testing `[K2]`:** Know that white-box techniques can be used in static testing (dry runs, pseudocode review).
