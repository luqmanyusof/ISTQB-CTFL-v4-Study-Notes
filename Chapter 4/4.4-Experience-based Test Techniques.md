## 🎯 4.4.1 Error Guessing `[K2]`

*FL-4.4.1: Explain error guessing*

### Definition
**Error guessing** is a test technique used to anticipate the occurrence of errors, defects, and failures based on the tester's knowledge.

### Knowledge Sources
- How the application has worked **in the past**
- Types of errors **developers tend to make** and resulting defects
- Types of failures that have occurred in **other, similar applications**

### Common Areas for Errors, Defects, and Failures

| Area | Examples |
|------|----------|
| **Input** | Correct input not accepted, parameters wrong or missing |
| **Output** | Wrong format, wrong result |
| **Logic** | Missing cases, wrong operator |
| **Computation** | Incorrect operand, wrong computation |
| **Interfaces** | Parameter mismatch, incompatible types |
| **Data** | Incorrect initialization, wrong type |

### Fault Attacks
A methodical way to implement error guessing:
1. Create or acquire a **list of possible errors, defects, and failures**
2. Design tests that will **identify defects**, expose the defects, or cause the failures
3. Build lists based on **experience, defect/failure data**, or common knowledge about why software fails

## 🔍 4.4.2 Exploratory Testing `[K2]`

*FL-4.4.2: Explain exploratory testing*

### Definition
In **exploratory testing**, tests are **simultaneously designed, executed, and evaluated** while the tester learns about the test object.

### Purpose
- Learn more about the test object
- Explore it more deeply with focused tests
- Create tests for untested areas

### Session-based Testing
A structured approach to exploratory testing:

| Component | Description |
|-----------|-------------|
| **Time Box** | Defined time period for the testing session |
| **Test Charter** | Contains test objectives to guide the testing |
| **Debriefing** | Discussion between tester and stakeholders about test results |
| **Test Session Sheets** | Document steps followed and discoveries made |

### When Exploratory Testing is Useful
- Few or **inadequate specifications**
- Significant **time pressure** on testing
- To **complement** other more formal test techniques

### Effectiveness Factors
Exploratory testing is more effective when the tester:
- Is **experienced**
- Has **domain knowledge**
- Has high degree of **essential skills** (analytical skills, curiosity, creativeness)

### Integration with Other Techniques
Exploratory testing can incorporate other test techniques (e.g., equivalence partitioning).

## ✅ 4.4.3 Checklist-based Testing `[K2]`

*FL-4.4.3: Explain checklist-based testing*

### Definition
In **checklist-based testing**, a tester designs, implements, and executes tests to cover test conditions from a checklist.

### Building Checklists
Checklists can be built based on:
- **Experience**
- Knowledge about **what is important for the user**
- Understanding of **why and how software fails**

### What Checklists Should NOT Contain
- Items that can be **checked automatically**
- Items better suited as **entry criteria or exit criteria**
- Items that are **too general**

### Checklist Item Characteristics
- Often phrased as a **question**
- Should be possible to check each item **separately and directly**
- May refer to requirements, graphical interface properties, quality characteristics, or other test conditions

### Types of Checklists
Can support various test types including:
- **Functional testing**
- **Non-functional testing** (e.g., 10 heuristics for usability testing)

### Checklist Maintenance
- Some entries may become **less effective over time** (developers learn to avoid same errors)
- **New entries** may need to be added for newly found high-severity defects
- Checklists should be **regularly updated** based on defect analysis
- Avoid letting checklist become **too long**

### Trade-offs

| Aspect | High-level Checklists | Detailed Test Cases |
|--------|----------------------|---------------------|
| **Coverage** | Potentially greater | More controlled |
| **Repeatability** | Less repeatable | More repeatable |
| **Consistency** | Some degree of consistency | High consistency |
| **Variability** | More variability in actual testing | Less variability |

## 🧩 Key Takeaways

| Technique | Key Characteristic | When to Use |
|-----------|-------------------|-------------|
| **Error Guessing** | Based on tester's knowledge of common errors | When experienced testers can anticipate likely defects |
| **Exploratory Testing** | Simultaneous design, execution, evaluation | When specs are inadequate or time is limited |
| **Checklist-based Testing** | Guided by predefined checklist | When consistency is needed without detailed test cases |

---

## 🎯 Exam Prep Tips

- **Error Guessing `[K2]`:** Know the knowledge sources (past behavior, developer errors, similar applications); understand fault attacks.
- **Exploratory Testing `[K2]`:** Know it's simultaneous design/execution/evaluation; understand session-based testing components (time box, charter, debriefing).
- **Checklist-based Testing `[K2]`:** Know what checklists should NOT contain; understand the need for regular updates.
- **Complementary `[K2]`:** All experience-based techniques complement black-box and white-box techniques.
- **Tester Skills `[K2]`:** Effectiveness depends heavily on tester's experience, domain knowledge, and essential skills.
