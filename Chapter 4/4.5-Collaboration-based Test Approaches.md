Unlike other test techniques that focus on **defect detection**, collaboration-based approaches focus on **defect avoidance** through collaboration and communication.

## 📝 4.5.1 Collaborative User Story Writing `[K2]`

*FL-4.5.1: Explain how to write user stories in collaboration with developers and business representatives*

### Definition
A **user story** represents a feature that will be valuable to either a user or purchaser of a system or software.

### The 3 C's of User Stories

| Component | Description |
|-----------|-------------|
| **Card** | The medium describing a user story (e.g., index card, electronic board entry) |
| **Conversation** | Explains how the software will be used (can be documented or verbal) |
| **Confirmation** | The acceptance criteria |

### User Story Format
```
As a [role], 
I want [goal to be accomplished], 
so that I can [resulting business value for the role]
```
Followed by the **acceptance criteria**.

### Collaborative Authorship
- Use techniques such as **brainstorming** and **mind mapping**
- Collaboration allows the team to obtain a **shared vision** of what should be delivered
- Takes into account **three perspectives:**
  - Business
  - Development
  - Testing

### INVEST Criteria for Good User Stories

| Letter | Criterion | Description |
|--------|-----------|-------------|
| **I** | Independent | Can be developed and tested independently |
| **N** | Negotiable | Details can be negotiated |
| **V** | Valuable | Provides value to stakeholders |
| **E** | Estimable | Can be estimated for effort |
| **S** | Small | Small enough to complete in one iteration |
| **T** | Testable | Can be verified through testing |

> If a stakeholder doesn't know how to test a user story, this may indicate the story is **not clear enough**, doesn't reflect something **valuable**, or the stakeholder needs **help in testing**.

## ✅ 4.5.2 Acceptance Criteria `[K2]`

*FL-4.5.2: Classify the different options for writing acceptance criteria*

### Definition
**Acceptance criteria** are the conditions that an implementation of the user story must meet to be accepted by stakeholders.

### Purpose of Acceptance Criteria
- **Define the scope** of the user story
- **Reach consensus** among stakeholders
- **Describe both positive and negative scenarios**
- **Serve as a basis** for user story acceptance testing
- **Allow accurate planning and estimation**

### Two Common Formats

| Format | Description | Example |
|--------|-------------|---------|
| **Scenario-oriented** | Given/When/Then format (used in Behavior-Driven Development (BDD)) | Given [precondition], When [action], Then [expected result] |
| **Rule-oriented** | Bullet point verification list or tabulated input-output mapping | List of conditions that must be true |

### Key Points
- Acceptance criteria are usually a result of the **Conversation** (from the 3 C's)
- May be viewed as **test conditions** that should be exercised by tests
- Team may use **custom formats** as long as criteria are well-defined and unambiguous

## 🚀 4.5.3 Acceptance Test-Driven Development (ATDD) `[K3]`

*FL-4.5.3: Use acceptance test-driven development (ATDD) to derive test cases*

### Definition
**Acceptance Test-Driven Development (ATDD)** is a test-first approach where test cases are created **prior to implementing** the user story.

### Key Characteristics
- Test cases created by team members with **different perspectives** (customers, developers, testers)
- Test cases may be executed **manually or automated**

### ATDD Process

| Step | Activity | Description |
|------|----------|-------------|
| 1 | **Specification Workshop** | User story and acceptance criteria are analyzed, discussed, and written by team members |
| 2 | **Resolve Issues** | Incompleteness, ambiguities, or defects in user story are resolved |
| 3 | **Create Test Cases** | Done by team as a whole or by tester individually; based on acceptance criteria |
| 4 | **Implement User Story** | Test cases serve as examples of how software works; helps team implement correctly |

### Test Case Creation Order
1. **Positive test cases first:** Confirm correct behavior without exceptions or error conditions
2. **Negative testing:** Test error conditions and exceptions
3. **Non-functional testing:** Cover quality characteristics (e.g., performance efficiency, usability)

### Test Case Characteristics
- Expressed in a way **understandable for stakeholders**
- Typically contain sentences in **natural language**
- Include: preconditions (if any), inputs, and postconditions
- Must cover **all characteristics** of the user story
- Should **not go beyond** the story
- **No two test cases** should describe the same characteristics

### Test Techniques in ATDD
During test design, techniques from sections 4.2, 4.3, and 4.4 may be applied:
- Black-box techniques (EP, BVA, Decision Tables, State Transition)
- White-box techniques (Statement, Branch)
- Experience-based techniques (Error Guessing, Exploratory, Checklist-based)

### Automation
When captured in a format supported by a test automation framework:
- Developers can **automate test cases** by writing supporting code
- Acceptance tests become **executable requirements**

## 🧩 Key Takeaways

| Concept | Key Point |
|---------|----------|
| **User Stories** | 3 C's: Card, Conversation, Confirmation; INVEST criteria |
| **Acceptance Criteria** | Scenario-oriented (Given/When/Then) or Rule-oriented formats |
| **ATDD** | Test-first approach; specification workshop → create tests → implement |
| **Collaboration** | Three perspectives: Business, Development, Testing |

- **User Stories `[K2]`:** Know the 3 C's (Card, Conversation, Confirmation) and INVEST criteria.
- **User Story Format `[K2]`:** Know the "As a [role], I want [goal], so that [value]" format.
- **Acceptance Criteria `[K2]`:** Know the two formats: scenario-oriented (Given/When/Then) and rule-oriented.
- **ATDD `[K3]`:** Be prepared to APPLY ATDD to derive test cases; know the process steps.
- **Test Order `[K3]`:** Positive tests first, then negative tests, then non-functional tests.
- **Defect Avoidance `[K2]`:** Collaboration-based approaches focus on defect avoidance, not just detection.
