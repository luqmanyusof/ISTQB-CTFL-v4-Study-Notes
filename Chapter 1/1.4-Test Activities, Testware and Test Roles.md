## 🔑 1.4.1 Test Activities and Tasks `[K2]`

A test process usually consists of the main groups of activities below. Although many appear to follow a logical sequence, they are often implemented **iteratively or in parallel**.

### 1. Test Planning
- **Purpose:** Define test objectives and select an approach that best achieves objectives within constraints.
- **Activities:** Defining scope, objectives, approach, resources, schedule, entry/exit criteria.
- **Deliverables:** Test plan, test schedule, risk register, entry/exit criteria.
- **Key Point:** Test planning is context-dependent and tailored to the specific situation.

---

### 2. Test Monitoring and Control
- **Test Monitoring:** Ongoing checking of all test activities and comparison of actual progress against the plan.
- **Test Control:** Taking actions necessary to meet the test objectives when deviations occur.
- **Deliverables:** Test progress reports, control directives, risk information updates.
- **Key Point:** Monitoring = observe; Control = act.

### 3. Test Analysis
- **Purpose:** Identify **"what to test"** in terms of measurable coverage criteria.
- **Activities:** Analyzing test basis to identify testable features; defining and prioritizing test conditions based on risks.
- **Inputs:** Test basis (requirements, design docs, code, risk analysis).
- **Outputs:** Prioritized test conditions, defect reports from static testing.
- **Key Point:** Test analysis is often supported by test techniques (see Chapter 4).

---

### 4. Test Design
- **Purpose:** Define **"how to test"** by elaborating test conditions into test cases and other testware.
- **Activities:** Identifying coverage items, specifying test case inputs, designing test data requirements, designing test environment, identifying infrastructure and tools.
- **Outputs:** Test cases, test charters, coverage items, test data requirements, test environment requirements.
- **Key Point:** Test techniques can be used to support test design activities.

---

### 5. Test Implementation
- **Purpose:** Create or acquire testware necessary for test execution and prepare for execution.
- **Activities:** Organizing test cases into test procedures and test suites; creating manual and automated test scripts; prioritizing and arranging test execution schedule; building test environment.
- **Outputs:** Test procedures, automated test scripts, test suites, test data, test execution schedule, test environment elements (stubs, drivers, simulators, service virtualizations).
- **Key Point:** Test procedures are prioritized and arranged for efficient test execution.

---

### 6. Test Execution
- **Purpose:** Run tests in accordance with the test execution schedule (test runs).
- **Activities:** Running tests (manual or automated), comparing actual results with expected results, logging test results, analyzing anomalies to identify likely causes, reporting anomalies based on observed failures.
- **Outputs:** Test logs, defect reports.
- **Forms:** Continuous testing, pair testing sessions, automated test runs.
- **Key Point:** Actual test results are compared with expected results to determine pass/fail status.

---

### 7. Test Completion
- **Purpose:** Wrap up test activities and learn from the testing performed.
- **Occurs at:** Project milestones (release, end of iteration, test level completion).
- **Activities:** Creating change requests for unresolved defects; identifying and archiving useful testware; shutting down test environment; analyzing test activities for lessons learned; creating test completion report.
- **Outputs:** Test completion report, action items for improvement, documented lessons learned, change requests.
- **Key Point:** Lessons learned are used to improve future iterations, releases, or projects.

## 📂 1.4.3 Testware (Work Products) `[K2]`

Testware is created as output work products from test activities. Proper **configuration management** ensures consistency and integrity.

| Test Activity | Work Products |
|---------------|---------------|
| **Planning** | Test plan, test schedule, risk register, entry/exit criteria |
| **Monitoring & Control** | Test progress reports, control directives, risk information |
| **Analysis** | Prioritized test conditions, defect reports (from static testing) |
| **Design** | Test cases, test charters, coverage items, test data/environment requirements |
| **Implementation** | Test procedures, automated test scripts, test suites, test data, test execution schedule, test environment elements (stubs, drivers, simulators) |
| **Execution** | Test logs, defect reports |
| **Completion** | Test completion report, action items, documented lessons learned, change requests |

**Key Point:** There is significant variation in how organizations produce, name, organize, and manage testware.

## 📊 1.4.2 Test Process in Context `[K2]`

Testing is NOT performed in isolation. Test activities are an integral part of development processes. The way testing is carried out depends on contextual factors:

- **Stakeholders:** Needs, expectations, requirements, willingness to cooperate.
- **Team members:** Skills, knowledge, experience level, availability, training needs.
- **Business domain:** Criticality of test object, identified risks, market needs, legal regulations.
- **Technical factors:** Type of software, product architecture, technology used.
- **Project constraints:** Scope, time, budget, resources available.
- **Organizational factors:** Organizational structure, existing policies, practices used.
- **SDLC:** Engineering practices, development methods being followed.
- **Tools:** Availability, usability, compliance requirements.

**Impact on Testing:** These factors affect test strategy, test techniques, degree of automation, required coverage level, level of detail of testware, and test reporting.

## 🔗 1.4.4 Traceability `[K2]`

To implement effective test monitoring and control, establish and maintain **traceability throughout the test process** between test basis elements, testware, test results, and defects.

**Benefits of Good Traceability:**
- **Coverage evaluation:** Verify that requirements are covered by test cases.
- **Risk assessment:** Evaluate the level of residual risk in a test object.
- **Impact analysis:** Determine the impact of changes on testing.
- **Audit support:** Meet Information Technology (IT) governance criteria.
- **Understandable reports:** Make test progress and completion reports more easily understandable.
- **Stakeholder communication:** Assist in communicating technical aspects of testing in an understandable manner.

**Examples:**
- Requirement Traceability Matrix (RTM): Links requirements → test cases.
- Risk Traceability: Links risks → test results to assess residual risk.

**Key Point:** Accurate traceability supports coverage evaluation, so it is useful if measurable coverage criteria are defined in the test basis.

## 👥 1.4.5 Roles in Testing `[K2]`

Two principal roles are covered in this syllabus:

| Role | Focus | Main Activities |
|------|-------|----------------|
| **Test Management Role** | Process & coordination | Test planning, test monitoring, test control, test completion |
| **Testing Role** | Technical/engineering aspect | Test analysis, test design, test implementation, test execution |

**Key Points:**
- The way roles are carried out varies depending on **context** (project, product, skills, organization).
- In **Agile software development**, some test management tasks may be handled by the Agile team itself.
- Tasks spanning multiple teams or the entire organization may be performed by **test managers outside the development team**.
- Different people may take on these roles at **different times** (e.g., team leader, test manager, development manager).
- It is possible for **one person to take on both roles** at the same time.

## 🎯 Exam Prep Tips

- **Test Activities `[K2]`:** Know all 7 activities and their purposes — they may appear to be sequential but are often iterative/parallel.
- **Analysis vs. Design `[K2]`:** Analysis = "what to test"; Design = "how to test" — common exam distinction.
- **Design vs. Implementation `[K2]`:** Design = identify requirements; Implementation = set them up — another common trap.
- **Monitoring vs. Control `[K2]`:** Monitoring = observe progress; Control = take corrective action.
- **Context Factors `[K2]`:** Know the factors that influence how testing is carried out (stakeholders, team, domain, technical, constraints, organizational, SDLC, tools).
- **Testware `[K2]`:** Know which work products are produced by each test activity.
- **Traceability Benefits `[K2]`:** Coverage evaluation, impact analysis, audit support, understandable reports.
- **Roles `[K2]`:** Test management (process) vs. Testing (technical) — know the main activities for each.
