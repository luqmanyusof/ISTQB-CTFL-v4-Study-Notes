### 🔎 Definition
- **Software testing** is a set of activities to discover defects and evaluate the quality of software work products.
- These work products, when being tested, are known as **test objects** (the item being tested).
- Testing is NOT just executing tests — it includes planning, analysis, design, implementation, reporting, and evaluation.
- Testing must be aligned with the **Software Development Lifecycle (SDLC)** to be effective.

---

### 🧠 Why Testing is Needed
- Software that does not work correctly can lead to **loss of money, time, business reputation**, and in extreme cases, **injury or death**.
- Testing assesses software quality and helps **reduce the risk of software failure** in operation.
- A common misconception: testing only consists of executing tests — but it also includes many other activities.

**Verification vs. Validation:**
- **Verification:** Checking whether the system meets **specified requirements** (building the product right).
- **Validation:** Checking whether the system meets **users' and stakeholders' needs** in its operational environment (building the right product).

**Static vs. Dynamic Testing:**
- **Dynamic testing:** Involves the **execution of software** to find defects and cause failures.
- **Static testing:** Does NOT involve execution — includes **reviews and static analysis** of work products.

### 🎯 1.1.1 Test Objectives `[K1]`

*FL-1.1.1: Identify typical test objectives*

The typical test objectives are:
- **Evaluating work products:** Assess requirements, user stories, designs, and code for quality and completeness.
- **Causing failures and finding defects:** Trigger failures through execution to identify underlying defects in the software.
- **Ensuring required coverage:** Verify that the test object is adequately covered by test cases.
- **Reducing risk level:** Lower the probability of inadequate software quality reaching production environments.
- **Verifying requirements:** Confirm that specified requirements have been fulfilled as documented.
- **Compliance verification:** Ensure the test object complies with contractual, legal, and regulatory requirements.
- **Providing information:** Give stakeholders data to allow them to make informed decisions about releases.
- **Building confidence:** Establish trust in the quality of the test object before deployment.
- **Validating completeness:** Confirm the test object is complete and works as expected by stakeholders.

> Test objectives vary depending on context: work product being tested, test level, risks, SDLC, and business factors.

### ⚖️ 1.1.2 Testing and Debugging `[K2]`

*FL-1.1.2: Differentiate testing from debugging*

| Activity | Purpose | Performed By |
|----------|---------|-------------|
| **Testing** | Triggers failures (dynamic) or finds defects directly (static) | Testers |
| **Debugging** | Finds causes of failures, analyzes them, and eliminates defects | Developers |

**The Debugging Process (after dynamic testing triggers a failure):**
1. **Reproduction:** Recreate the failure to understand when and how it occurs.
2. **Diagnosis:** Find the defect (root cause) that caused the failure.
3. **Fixing:** Eliminate the defect by modifying the code or configuration.

**After Debugging:**
- **Confirmation testing:** Checks whether the fix resolved the problem (preferably by the same tester).
- **Regression testing:** Checks whether the fix caused failures in other parts of the system.

> When static testing identifies a defect, there is no need for reproduction or diagnosis — static testing finds defects directly.

---

### 👥 Testing is an Intellectual Activity
- Testing is **NOT only a technical activity** — it needs proper planning, management, estimation, monitoring, and control.
- Testers use tools, but testing is largely an **intellectual activity** requiring:
  - **Specialized knowledge:** Understanding of testing concepts, techniques, and domain.
  - **Analytical skills:** Ability to analyze requirements, identify risks, and design effective tests.
  - **Critical thinking:** Questioning assumptions and challenging the system under test.
  - **Systems thinking:** Understanding how components interact within the larger system.

> The International Organization for Standardization/International Electrotechnical Commission/Institute of Electrical and Electronics Engineers (ISO/IEC/IEEE) 29119-1 standard provides further information about software testing concepts.

### 🎯 Exam Prep Tips

- **Test Objectives `[K1]`:** Be able to identify and recall the typical test objectives from the syllabus list.
- **Testing vs. Debugging `[K2]`:** Clearly differentiate — testing finds/triggers defects; debugging fixes them.
- **Verification vs. Validation:** Verification = meets requirements; Validation = meets user needs.
- **Static vs. Dynamic:** Static = no execution (reviews); Dynamic = execution required.
- **Confirmation vs. Regression:** Confirmation = fix works; Regression = no side effects from fix.
