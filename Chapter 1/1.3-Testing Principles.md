# 📘 ISTQB CTFL v4.0 – Study Notes
## Chapter 1.3: Testing Principles `[K2]`

*FL-1.3.1: Explain the seven testing principles*

A number of testing principles offering general guidelines applicable to all testing have been suggested over the years. This syllabus describes **seven such principles**.

---

### 1️⃣ Testing shows the presence, not the absence of defects
- **Principle:** Testing can show that defects are present in the test object, but **cannot prove that there are no defects**.
- **Key Point:** Testing reduces the probability of defects remaining undiscovered, but even if no defects are found, **testing cannot prove test object correctness**.
- **Implication:** You can never claim software is "bug-free" — only that no bugs were found during testing.
- **Reference:** Buxton 1970

---

### 2️⃣ Exhaustive testing is impossible
- **Principle:** Testing everything (all combinations of inputs and preconditions) is **not feasible except in trivial cases**.
- **Key Point:** Rather than attempting to test exhaustively, use **test techniques, test case prioritization, and risk-based testing** to focus test efforts.
- **Implication:** Smart test design is essential — you must prioritize what to test based on risk and importance.
- **Reference:** Manna 1978

---

### 3️⃣ Early testing saves time and money
- **Principle:** Defects that are removed early in the process **will not cause subsequent defects** in derived work products.
- **Key Point:** The **cost of quality will be reduced** since fewer failures will occur later in the SDLC.
- **Implication:** Both static testing (reviews) and dynamic testing should be started **as early as possible**.
- **Reference:** Boehm 1981

---

### 4️⃣ Defects cluster together
- **Principle:** A small number of system components usually contain **most of the defects** discovered or are responsible for most operational failures.
- **Key Point:** This phenomenon illustrates the **Pareto principle (80/20 rule)** — 80% of defects come from 20% of modules.
- **Implication:** Predicted defect clusters and actual observed clusters are important input for **risk-based testing**.
- **Reference:** Enders 1975

---

### 5️⃣ Tests wear out (Pesticide Paradox)
- **Principle:** If the same tests are repeated many times, they become **increasingly ineffective** in detecting new defects.
- **Key Point:** To overcome this effect, existing tests and test data may need to be **modified**, and **new tests may need to be written**.
- **Exception:** In some cases, repeating the same tests can have beneficial outcomes, e.g., in **automated regression testing**.
- **Reference:** Beizer 1990

---

### 6️⃣ Testing is context-dependent
- **Principle:** There is **no single universally applicable approach** to testing.
- **Key Point:** Testing is done **differently in different contexts** — what works for one project may not work for another.
- **Implication:** Test strategy must be adapted based on system type, domain, risks, regulations, and stakeholder needs.
- **Reference:** Kaner 2011

---

### 7️⃣ Absence-of-defects fallacy
- **Principle:** It is a **fallacy (misconception)** to expect that software verification will ensure the success of a system.
- **Key Point:** Thoroughly testing all specified requirements and fixing all defects found could still produce a system that:
  - Does **not fulfill users' needs and expectations**
  - Does **not help achieve customer's business goals**
  - Is **inferior compared to competing systems**
- **Implication:** In addition to **verification**, **validation** should also be carried out.
- **Reference:** Boehm 1981

---

## 🎯 Exam Prep Tips

- **All 7 Principles `[K2]`:** Be able to explain each principle and provide examples of how they apply.
- **Principle 1:** Testing proves presence of defects, NOT their absence — never claim "bug-free."
- **Principle 2:** Exhaustive testing is impossible — use risk-based testing and prioritization.
- **Principle 3:** Early testing saves time and money — shift left approach.
- **Principle 4:** Defects cluster together — Pareto principle (80/20 rule).
- **Principle 5:** Tests wear out (Pesticide Paradox) — update and add new tests regularly.
- **Principle 6:** Testing is context-dependent — no universal approach exists.
- **Principle 7:** Absence-of-defects fallacy — verification alone is not enough; validation is also needed.
