## 🔎 What is Static Testing?

**Static testing** = evaluating software work products **without executing the code**.

| Method | Description |
|--------|-------------|
| **Manual Examination (Reviews)** | Human reviewers examine work products to find defects and assess quality. |
| **Tool-based (Static Analysis)** | Automated tools analyze code or documents against predefined rules. |

**Test Objectives of Static Testing:**
- Improving quality of work products
- Detecting defects early in the Software Development Lifecycle (SDLC)
- Assessing characteristics like readability, completeness, correctness, testability, and consistency

> Static testing can be applied for both **verification** (building the product right) and **validation** (building the right product).

## 📄 3.1.1 Work Products Examinable by Static Testing `[K1]`

*FL-3.1.1: Recognize types of work products that can be examined by static testing*

**Almost any work product can be examined using static testing:**

| Work Product Type | Examples |
|-------------------|----------|
| **Requirements** | Requirement specification documents, user stories, acceptance criteria |
| **Source Code** | Code files in any programming language |
| **Test Artifacts** | Test plans, test cases, test charters |
| **Design Documents** | System architecture specifications, process specifications |
| **Project Documentation** | Contracts, project plans, schedules |
| **Models** | UML diagrams, data models, workflow diagrams |
| **Product Backlog Items** | User stories with acceptance criteria |

### Requirements for Different Static Testing Methods

| Method | Requirement |
|--------|-------------|
| **Reviews** | Any work product that can be **read and understood** by humans |
| **Static Analysis** | Work products need a **formal structure** against which they can be checked (e.g., models, code, text with formal syntax) |

### Work Products NOT Appropriate for Static Testing

- **Third-party executable code:** Cannot be analyzed due to legal restrictions and lack of source access.
- **Work products difficult to interpret:** Items that humans cannot easily read or tools cannot parse effectively.

## 💎 3.1.2 Value of Static Testing `[K2]`

*FL-3.1.2: Explain the value of static testing*

### Early Defect Detection
- Detects defects in the **earliest phases of the SDLC**, fulfilling the principle of early testing.
- Identifies defects that **cannot be detected by dynamic testing**:
  - Unreachable code
  - Design patterns not implemented as desired
  - Defects in non-executable work products

### Quality and Confidence Building
- Evaluates the **quality of work products** and builds confidence before code execution begins.
- Verifies documented requirements actually describe **stakeholder's actual needs** accurately and completely.

### Improved Communication
- Creates **shared understanding** among stakeholders when performed early in the SDLC.
- Improves **communication between stakeholders** by involving a wide variety of participants in reviews.
- It is recommended to involve a **wide variety of stakeholders** in static testing.

### Cost Efficiency
- Reviews may be **costly to implement initially**, but overall project costs are usually much lower.
- Less time and effort spent on **fixing defects later** in the project when caught early.
- Static analysis detects code defects **more efficiently** than dynamic testing, reducing overall development effort.
- Results in both **fewer code defects** and a **lower overall development effort**.

## ⚖️ 3.1.3 Differences Between Static and Dynamic Testing `[K2]`

*FL-3.1.3: Compare and contrast static testing and dynamic testing*

### Comparison Table

| Aspect | Static Testing | Dynamic Testing |
|--------|----------------|-----------------|
| **Execution** | Software does **NOT** need to be executed | Software **MUST** be executed |
| **Defect Detection** | Finds defects **directly** in work products | Causes **failures**, then determines defects through analysis |
| **Work Products** | Can be applied to **non-executable** work products | Can only be applied to **executable** work products |
| **Rare Paths** | Easily detects defects on **rarely executed** code paths | Hard to reach paths that are rarely executed |
| **Quality Characteristics** | Measures characteristics **not dependent on execution** (e.g., maintainability) | Measures characteristics **dependent on execution** (e.g., performance efficiency) |

> Static and dynamic testing practices **complement each other** with similar objectives but different approaches.

### Defects Easier/Cheaper to Find Through Static Testing

| Defect Category | Examples |
|-----------------|----------|
| **Requirements Defects** | Inconsistencies, ambiguities, contradictions, omissions, inaccuracies, duplications |
| **Design Defects** | Inefficient database structures, poor modularization |
| **Coding Defects** | Variables with undefined values, undeclared variables, unreachable code, duplicated code, excessive complexity |
| **Standards Deviations** | Lack of adherence to naming conventions and coding standards |
| **Interface Specification Errors** | Mismatched number, type, or order of parameters |
| **Security Vulnerabilities** | Buffer overflows and other specific security issues |
| **Test Basis Coverage Gaps** | Missing tests for acceptance criteria |

## 🔧 Static Analysis in Practice

**Static analysis** can identify problems prior to dynamic testing while often requiring **less effort** since no test cases are required.

**Common Uses:**
- Incorporated into **Continuous Integration (CI) frameworks**
- Detecting specific **code defects**
- Evaluating **maintainability** and **security**
- **Spelling checkers** and **readability tools**

**Collaborative Activities Using Static Testing:**
- Example mappings
- Collaborative user story writing
- Backlog refinement sessions
- Ensuring user stories meet **Definition of Ready** criteria

## 🎯 Exam Prep Tips

- **Work Products `[K1]`:** Remember almost any readable work product can be reviewed; static analysis needs formal structure.
- **Value of Static Testing `[K2]`:** Emphasize early defect detection, cost savings, and improved stakeholder communication.
- **Static vs. Dynamic `[K2]`:** Know that static testing finds defects directly without execution; dynamic testing causes failures then analyzes them.
- **Defect Types `[K2]`:** Know which defect types are easier to find with static testing (requirements, design, coding, standards, interfaces, security).
- **Complementary `[K2]`:** Static and dynamic testing complement each other — both are needed for comprehensive testing.
- **Non-executable `[K1]`:** Static testing can examine non-executable work products; dynamic testing cannot.
