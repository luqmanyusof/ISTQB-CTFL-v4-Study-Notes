# Teaching Notes: 3.2 Feedback and Review Process

## Benefits of Early Feedback

| Benefit | Software Example |
|---------|------------------|
| Prevents Misunderstandings | Clarify "search" means full-text, not exact match |
| Focuses on Value | Prioritize payment feature over admin reports |
| Reduces Risk | Identify security concern before implementation |

---

## Review Process Activities

| Activity | Software Example |
|----------|------------------|
| Planning | Schedule code review for authentication module |
| Review Initiation | Share pull request link with reviewers |
| Individual Review | Each reviewer examines code independently |
| Communication & Analysis | Team discusses found issues in meeting |
| Fixing & Reporting | Developer fixes issues, updates pull request |

---

## Review Roles

| Role | Software Example |
|------|------------------|
| Manager | Allocates time for sprint review activities |
| Author | Developer who wrote the code being reviewed |
| Moderator | Facilitates review meeting, keeps discussion on track |
| Scribe | Documents all issues found during review |
| Reviewer | Examines code and identifies potential bugs |
| Review Leader | Organizes review, decides who participates |

---

## Review Types

| Type | Software Example |
|------|------------------|
| Informal | Quick desk check of colleague's code |
| Walkthrough | Author explains new API design to team |
| Technical Review | Senior developers review database schema |
| Inspection | Formal review with checklist and metrics |

---

## Success Factors for Reviews

| Factor | Software Example |
|--------|------------------|
| Clear Objectives | Goal: find security vulnerabilities in login |
| Manageable Scope | Review 200 lines, not 2000 lines |
| Adequate Prep Time | Give reviewers 1 day before meeting |
| No Blame | Focus on code quality, not developer skill |

---
