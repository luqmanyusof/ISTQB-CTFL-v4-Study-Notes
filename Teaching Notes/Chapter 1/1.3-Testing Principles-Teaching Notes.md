# Teaching Notes: 1.3 Testing Principles

![Testing Workflow](https://commons.wikimedia.org/wiki/File:Controller_Testing_Steps_Diagram.png)

## The 7 Testing Principles with Software Examples

### 1. Testing shows presence, not absence of defects
- **Concept:** Finding bugs proves they exist; not finding doesn't prove bug-free
- **Example:** Passing all tests doesn't guarantee zero bugs

### 2. Exhaustive testing is impossible
- **Concept:** Cannot test all input combinations
- **Example:** Password field has infinite possible character combinations

### 3. Early testing saves time and money
- **Concept:** Fix bugs early = cheaper than fixing later
- **Example:** Catching API design flaw before coding saves weeks

### 4. Defects cluster together
- **Concept:** Most bugs found in few modules (80/20 rule)
- **Example:** Payment module has 80% of reported bugs

### 5. Tests wear out (Pesticide Paradox)
- **Concept:** Same tests stop finding new bugs over time
- **Example:** Old regression suite misses new edge cases

### 6. Testing is context-dependent
- **Concept:** Different projects need different testing approaches
- **Example:** Banking app needs more security testing than game

### 7. Absence-of-defects fallacy
- **Concept:** Bug-free doesn't mean users will like it
- **Example:** Perfect app that nobody wants to use

## Quick Reference (≤10 words each)

| Principle | Software Example |
|-----------|------------------|
| Presence not absence | 1000 passed tests, still crashed in production |
| Exhaustive impossible | Search field: infinite query combinations to test |
| Early testing | Requirements review catches missing validation rule |
| Defects cluster | Legacy authentication module contains most bugs |
| Pesticide paradox | Add new tests after each release cycle |
| Context-dependent | Medical device needs stricter testing than blog |
| Absence-of-defects | Zero bugs but users hate the UI |
