## 🔎 What is an SDLC Model?

An **SDLC model** is an abstract, high-level representation of the software development process defining how phases and activities relate logically and chronologically.

**SDLC Model Examples:**
- **Sequential:** Waterfall, V-model
- **Iterative:** Spiral model, Prototyping
- **Incremental:** Unified Process
- **Agile Practices:** Acceptance Test-Driven Development (ATDD), Behavior-Driven Development (BDD), Domain-Driven Design (DDD), Extreme Programming (XP), Feature-Driven Development (FDD), Kanban, Lean IT, Scrum, Test-Driven Development (TDD)

## 🏗️ 2.1.1 Impact of the SDLC on Testing `[K2]`

*FL-2.1.1: Explain the impact of the chosen software development lifecycle on testing*

Testing must adapt to the SDLC. The choice impacts:
- **Scope and timing:** Determines when testing occurs and which test levels and types are applied throughout development phases.
- **Level of detail:** Defines how comprehensive test documentation should be, from lightweight notes to formal specifications.
- **Choice of test techniques:** Influences whether to use exploratory, scripted, or risk-based testing approaches for the project.
- **Extent of test automation:** Dictates how much automation is feasible based on iteration speed and regression needs.
- **Role and responsibilities:** Shapes whether testers work independently or are embedded within cross-functional development teams.

| SDLC Type | Testing Characteristics |
|-----------|------------------------|
| **Sequential** | Testers participate in requirement reviews, test analysis/design early; dynamic testing later when code exists |
| **Iterative/Incremental** | Each iteration delivers working product; static and dynamic testing at all levels; frequent regression testing needed |
| **Agile** | Lightweight documentation; extensive test automation; experience-based techniques favored |

## ✅ 2.1.2 Good Testing Practices (All SDLCs) `[K1]`

*FL-2.1.2: Recall good testing practices that apply to all software development lifecycles*

- **Correspondence:** Every development activity has a corresponding test activity to ensure quality is verified at each stage.
- **Unique Objectives:** Each test level has specific goals to avoid redundancy and ensure comprehensive coverage without duplication.
- **Early Analysis:** Test analysis and design begins during corresponding development phase, applying the early testing principle effectively.
- **Early Review:** Testers review work products as soon as drafts are available, supporting shift left and catching defects early.

## 🚀 2.1.3 Testing as a Driver for Software Development `[K1]`

*FL-2.1.3: Recall the examples of test-first approaches to development*

Test-first approaches implement early testing and shift left principles:

| Approach | Description |
|----------|-------------|
| **Test-Driven Development (TDD)** | Write test → see it fail → write minimum code to pass → refactor |
| **Acceptance Test-Driven Development (ATDD)** | Derive tests from acceptance criteria before development |
| **Behavior-Driven Development (BDD)** | Express behavior using **Given/When/Then** format; auto-translate to executable tests |

All approaches produce automated tests that persist for future regression testing.

## ♾️ 2.1.4 DevOps and Testing `[K2]`

*FL-2.1.4: Summarize how DevOps might have an impact on testing*

**DevOps** = organizational approach creating synergy between development (including testing) and operations.

**Benefits for Testing:**
- **Fast feedback:** Developers receive immediate information on code quality and any adverse effects from recent changes.
- **CI promotes shift left:** Developers submit high-quality code with component tests, catching issues before integration occurs.
- **CI/CD stable environments:** Automated pipelines create consistent and reliable test environments for repeatable testing results.
- **Non-functional visibility:** Performance, reliability, and other quality characteristics become measurable and trackable throughout development.
- **Reduced manual testing:** Automation handles repetitive test execution, freeing testers for exploratory and creative testing work.
- **Minimized regression risk:** Automated regression tests run frequently to detect unintended side effects from code changes.

**Risks/Challenges:**
- **Pipeline definition:** DevOps delivery pipeline must be carefully designed, established, and aligned with testing requirements.
- **Tool maintenance:** Continuous Integration/Continuous Delivery (CI/CD) tools require ongoing introduction, configuration, updates, and maintenance by skilled personnel.
- **Automation resources:** Test automation demands additional investment in tools, frameworks, skills, and dedicated team members.

> Manual testing (especially user perspective) remains essential even with DevOps.

## ⬅️ 2.1.5 Shift Left `[K2]`

*FL-2.1.5: Explain shift left*

**Shift Left** = performing testing earlier in the SDLC. Does NOT mean neglecting later testing.

**Good Practices:**
- **Review specifications:** Examine requirements from tester's perspective to find ambiguities, incompleteness, and inconsistencies early.
- **Write tests before code:** Create test cases before implementation; run code in test harness during development.
- **Use CI/CD pipelines:** Leverage automated pipelines with fast feedback loops and automated component tests for rapid validation.
- **Static analysis first:** Complete static code analysis before dynamic testing to catch structural issues without execution.
- **Early non-functional testing:** Perform performance, security, and other non-functional testing starting at the component level.

**Trade-off:** Extra training/effort/cost early → saves effort/cost later

## 🔄 2.1.6 Retrospectives and Process Improvement `[K2]`

*FL-2.1.6: Explain how retrospectives can be used as a mechanism for process improvement*

**Retrospectives** are held at project/iteration end, release milestones, or as needed.

**Discussion Points:**
- **What was successful:** Identify practices and approaches that worked well and should be retained for future iterations.
- **What needs improvement:** Recognize areas that did not work well and discuss concrete ways to improve them.
- **How to incorporate:** Define actionable steps to implement improvements and assign ownership for follow-through.

**Benefits for Testing:**
- **Increased effectiveness/efficiency:** Testing processes become more streamlined and produce better results with less wasted effort.
- **Increased testware quality:** Test cases, scripts, and documentation improve based on lessons learned from previous cycles.
- **Team bonding and learning:** Shared reflection builds trust, knowledge transfer, and collective ownership of quality outcomes.
- **Improved test basis:** Requirements and specifications become clearer as feedback loops identify recurring documentation issues.
- **Better cooperation:** Development and testing teams align on goals, reducing friction and improving collaborative problem-solving.
