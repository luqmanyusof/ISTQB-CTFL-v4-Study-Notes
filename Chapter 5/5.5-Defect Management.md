## 🔎 What is Defect Management?

Since one of the major test objectives is to find defects, an **established defect management process** is essential.

### Key Points
- Reported **anomalies** may turn out to be real defects or something else (e.g., false-positive result, change request)
- This is resolved during the process of dealing with defect reports
- Anomalies may be reported during **any phase of the SDLC**
- Defects from **static testing** (especially static analysis) should be handled similarly

---

## 📋 Defect Management Process

### Minimum Requirements
- **Workflow** for handling individual defects/anomalies from discovery to closure
- **Rules for classification** of defects

### Workflow Activities
1. **Log** the reported anomalies
2. **Analyze and classify** them
3. **Decide** on suitable response (fix or keep as is)
4. **Close** the defect report

> The process must be followed by **all involved stakeholders**.

## 📝 5.5.1 Defect Report `[K3]`

*FL-5.5.1: Prepare a defect report*

### Objectives of Defect Reports
- Provide those responsible for handling and resolving defects with **sufficient information to resolve the issue**
- Provide a means of **tracking the quality** of the work product
- Provide ideas for **improvement** of the development and test process

### Typical Defect Report Content

| Field | Description |
|-------|-------------|
| **Unique Identifier** | Unique ID for tracking |
| **Title** | Short summary of the anomaly being reported |
| **Date** | When the anomaly was observed |
| **Issuing Organization** | Organization reporting the defect |
| **Author** | Person reporting, including their role |
| **Test Object Identification** | Which test object contains the defect |
| **Test Environment** | Environment where defect was found |
| **Context** | Test case being run, test activity being performed, SDLC phase, test technique, checklist or test data being used |
| **Description of Failure** | Steps to reproduce, test logs, database dumps, screenshots, recordings |
| **Expected Results** | What should have happened |
| **Actual Results** | What actually happened |
| **Severity** | Degree of impact on stakeholder interests or requirements |
| **Priority** | Urgency to fix |
| **Status** | Current state (e.g., open, deferred, duplicate, waiting to be fixed, awaiting confirmation testing, re-opened, closed, rejected) |
| **References** | Links to related test cases, requirements, etc. |

### Defect Status Values

| Status | Description |
|--------|-------------|
| **Open** | Newly reported, not yet addressed |
| **Deferred** | Will be addressed in future release |
| **Duplicate** | Same as another reported defect |
| **Waiting to be Fixed** | Assigned for fixing |
| **Awaiting Confirmation Testing** | Fix implemented, needs verification |
| **Re-opened** | Fix did not resolve the issue |
| **Closed** | Defect verified as fixed |
| **Rejected** | Not a valid defect |

### Severity vs. Priority

| Attribute | Definition | Example |
|-----------|------------|---------|
| **Severity** | Degree of **impact** on stakeholders or requirements | Critical, Major, Minor, Trivial |
| **Priority** | **Urgency** to fix | High, Medium, Low |

> Some data may be **automatically included** when using defect management tools (e.g., identifier, date, author, initial status).

### Reference
Document templates for defect reports can be found in the **ISO/IEC/IEEE 29119-3** standard (referred to as "incident reports").

## 🎯 Exam Prep Tips

- **Defect Report Content `[K3]`:** Be able to PREPARE a defect report; know all the typical fields.
- **Objectives `[K3]`:** Know the three objectives (provide info to resolve, track quality, provide improvement ideas).
- **Severity vs. Priority `[K3]`:** Severity = impact; Priority = urgency to fix.
- **Status Values `[K3]`:** Know the common status values (open, deferred, duplicate, waiting to be fixed, awaiting confirmation testing, re-opened, closed, rejected).
- **Workflow `[K3]`:** Know the workflow activities (log → analyze/classify → decide → close).
- **Static Testing `[K3]`:** Defects from static testing should be handled similarly.
