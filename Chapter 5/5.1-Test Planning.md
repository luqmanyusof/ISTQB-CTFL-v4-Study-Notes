## 📋 5.1.1 Purpose and Content of a Test Plan `[K2]`

*FL-5.1.1: Exemplify the purpose and content of a test plan*

### Definition
A **test plan** describes the test objectives, resources, and processes for a test project.

### Purpose of a Test Plan
- Documents the **means and schedule** for achieving test objectives
- Helps ensure test activities will **meet established criteria**
- Serves as a **means of communication** with team members and stakeholders
- Demonstrates testing will adhere to **test policy and test strategy** (or explains deviations)

### Value of Test Planning
- Forces testers to **confront future challenges** (risks, schedules, people, tools, costs, effort)
- Provides a useful way to **think through efforts** needed to achieve test objectives

### Typical Test Plan Content

| Category | Content |
|----------|---------|
| **Context** | Test scope, test objectives, test basis |
| **Assumptions & Constraints** | Limitations and dependencies of the test project |
| **Stakeholders** | Roles, responsibilities, relevance to testing, hiring/training needs |
| **Communication** | Forms and frequency of communication, documentation templates |
| **Risk Register** | Product risks, project risks |
| **Test Approach** | Test levels, test types, test techniques, test deliverables, entry/exit criteria, independence of testing, metrics, test data requirements, test environment requirements, deviations from policy/strategy |
| **Budget & Schedule** | Resources and timeline |

> More details in ISO/IEC/IEEE 29119-3 standard.

## 👥 5.1.2 Tester's Contribution to Iteration and Release Planning `[K1]`

*FL-5.1.2: Recognize how a tester adds value to iteration and release planning*

### Two Types of Planning in Iterative SDLCs

| Planning Type | Focus | Tester Contributions |
|---------------|-------|---------------------|
| **Release Planning** | Looks ahead to product release; defines product backlog | Write testable user stories and acceptance criteria; participate in risk analyses; estimate test effort; determine test approach; plan testing for release |
| **Iteration Planning** | Looks ahead to end of single iteration; concerned with iteration backlog | Detailed risk analysis of user stories; determine testability; break down user stories into tasks; estimate test effort for all testing tasks; identify functional and non-functional aspects |

## 🚪 5.1.3 Entry Criteria and Exit Criteria `[K2]`

*FL-5.1.3: Compare and contrast entry criteria and exit criteria*

### Definitions

| Criteria Type | Definition |
|---------------|------------|
| **Entry Criteria** | Preconditions for undertaking a given activity |
| **Exit Criteria** | What must be achieved to declare an activity completed |

> Entry and exit criteria should be defined for **each test level** and will differ based on test objectives.

### Typical Entry Criteria
- **Availability of resources:** People, tools, environments, test data, budget, time
- **Availability of testware:** Test basis, testable requirements, user stories, test cases
- **Initial quality level:** All smoke tests have passed

### Typical Exit Criteria
- **Measures of thoroughness:** Achieved coverage level, number of unresolved defects, defect density, number of failed test cases
- **Binary "yes/no" criteria:** Planned tests executed, static testing performed, all defects reported, all regression tests automated

### Special Cases
- **Running out of time or budget** can be valid exit criteria
- Acceptable to end testing if stakeholders have **reviewed and accepted the risk** to go live

### Agile Terminology

| ISTQB Term | Agile Term |
|------------|------------|
| Exit Criteria | **Definition of Done** (objective metrics for releasable item) |
| Entry Criteria | **Definition of Ready** (what user story must fulfill to start) |

## 📊 5.1.4 Estimation Techniques `[K3]`

*FL-5.1.4: Use estimation techniques to calculate the required test effort*

### Key Principles
- Estimates are based on **assumptions** and always subject to **estimation error**
- Estimation for **small tasks** is usually more accurate than for large ones
- **Decompose large tasks** into smaller tasks for better estimates

### Four Estimation Techniques

| Technique | Type | Description |
|-----------|------|-------------|
| **Estimation based on ratios** | Metrics-based | Use historical data from previous projects to derive standard ratios (e.g., dev:test = 3:2) |
| **Extrapolation** | Metrics-based | Measure early in project, then extrapolate for remaining work; suitable for iterative SDLCs |
| **Wideband Delphi** | Expert-based | Experts estimate in isolation → discuss deviations → re-estimate → repeat until consensus |
| **Three-point estimation** | Expert-based | Calculate weighted average of optimistic (a), most likely (m), and pessimistic (b) estimates |

### Three-point Estimation Formula
```
E = (a + 4*m + b) / 6
SD = (b - a) / 6
```

**Example:** If a=6, m=9, b=18 person-hours:
- E = (6 + 4×9 + 18) / 6 = **10 person-hours**
- SD = (18 - 6) / 6 = **±2 person-hours**
- Final estimate: **10±2 person-hours** (between 8 and 12)

### Planning Poker
A variant of Wideband Delphi commonly used in Agile; estimates made using cards with numbers representing effort size.

## 🎯 5.1.5 Test Case Prioritization `[K3]`

*FL-5.1.5: Apply test case prioritization*

### Prioritization Strategies

| Strategy | Basis | Description |
|----------|-------|-------------|
| **Risk-based** | Risk analysis results | Test cases covering most important risks executed first |
| **Coverage-based** | Coverage (e.g., statement) | Test cases achieving highest coverage executed first |
| **Requirements-based** | Requirement priorities | Test cases related to most important requirements executed first |

### Additional Coverage Prioritization
A variant where:
1. Execute test case with highest coverage first
2. Each subsequent test case achieves highest **additional** coverage

### Considerations
- **Dependencies:** If high-priority test case depends on low-priority one, execute low-priority first
- **Resource availability:** Test tools, environments, or people may only be available for specific time windows

## 🔺 5.1.6 Test Pyramid `[K1]`

*FL-5.1.6: Recall the concepts of the test pyramid*

### Definition
The **test pyramid** is a model showing that different tests may have different granularity.

### Purpose
- Supports team in **test automation** and **test effort allocation**
- Shows different test objectives are supported by **different levels of test automation**

### Pyramid Characteristics

| Layer | Granularity | Isolation | Execution Time | Coverage | Quantity |
|-------|-------------|-----------|----------------|----------|----------|
| **Bottom** (e.g., Unit) | High (small) | High (isolated) | Fast | Small piece of functionality | Many tests needed |
| **Top** (e.g., End-to-end) | Low (large) | Low (dependent) | Slow | Large piece of functionality | Few tests needed |

### Common Layer Names
- **Original model (Cohn 2009):** Unit tests → Service tests → UI tests
- **Alternative model:** Unit (component) tests → Integration (component integration) tests → End-to-end tests

## 🔲 5.1.7 Testing Quadrants `[K2]`

*FL-5.1.7: Summarize the testing quadrants and their relationships with test levels and test types*

### Definition
The **testing quadrants** (Brian Marick) group test levels with appropriate test types, activities, techniques, and work products in Agile software development.

### Two Dimensions
- **Business facing** vs. **Technology facing**
- **Support the team** (guide development) vs. **Critique the product** (measure behavior)

### The Four Quadrants

| Quadrant | Facing | Purpose | Test Types | Automation |
|----------|--------|---------|------------|------------|
| **Q1** | Technology | Support team | Component tests, Component integration tests | Automated, included in CI |
| **Q2** | Business | Support team | Functional tests, Examples, User story tests, User experience prototypes, API testing, Simulations | Manual or automated |
| **Q3** | Business | Critique product | Exploratory testing, Usability testing, User acceptance testing | User-oriented, often manual |
| **Q4** | Technology | Critique product | Smoke tests, Non-functional tests (except usability) | Often automated |

## 🎯 Exam Prep Tips

- **Test Plan Content `[K2]`:** Know the typical content categories (context, stakeholders, communication, risk register, test approach, budget/schedule).
- **Entry vs. Exit Criteria `[K2]`:** Entry = preconditions to start; Exit = conditions to complete; know Agile terms (Definition of Ready/Done).
- **Estimation Techniques `[K3]`:** Be able to APPLY all four techniques; know the three-point formula.
- **Prioritization `[K3]`:** Know the three strategies (risk-based, coverage-based, requirements-based).
- **Test Pyramid `[K1]`:** Recall the concept — bottom = many small fast tests; top = few large slow tests.
- **Testing Quadrants `[K2]`:** Know the four quadrants and their characteristics (Q1-Q4).
